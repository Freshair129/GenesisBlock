## 1. System Overview
GenesisDB is a high-performance, embedded graph database engine written in Rust. It provides the core storage and traversal capabilities for the Genesis Knowledge System (GKS). It exposes an asynchronous Node-API (FFI) interface for consumption by TypeScript environments.

## 2. Architecture

### 2.1 Core Subsystems
1.  **Storage Engine:** In-memory graph representation backed by a hybrid persistence model (JSONL WAL + Bincode snapshots).
2.  **Concurrency Manager:** Utilizes `parking_lot::RwLock` for high-throughput concurrent reads and exclusive writes.
3.  **Governance Layer:** Implements `Axiomatic Guards` to enforce tier-based hierarchical knowledge integrity during mutations.
4.  **Traversal Engine:** Provides BFS-based graph traversal algorithms optimized with visited-set tracking.
5.  **Impact Engine:** Calculates `$U(n)$` (Utility/Impact Score) for nodes incrementally to identify critical knowledge pathways.
6.  **Cypher Parser:** A lightweight regex-based engine parsing a subset of OpenCypher (`MATCH`, `WHERE`, `RETURN`).

### 2.2 Data Structures
*   **Nodes:** Mapped by `id (String) -> NodeOutput`.
*   **Edges:** Mapped by `id (String) -> EdgeOutput`.
*   **Indices:** HashMaps tracking `Outgoing` and `Incoming` edge IDs per Node ID for $O(1)$ adjacency lookups.

## 3. Data Model

### 3.1 `NodeInput` / `NodeOutput`
*   `id`: Unique identifier (String). Autogenerated via MD5 hash of contents if not provided.
*   `labels`: Classification tags (`Vec<String>`).
*   `props`: Arbitrary JSON payload (`serde_json::Value`).
*   `impact`: Engine-calculated utility score (`Option<f64>`).

### 3.2 `EdgeInput` / `EdgeOutput`
*   `id`: UUIDv4 string.
*   `from` / `to`: Source and Target Node IDs.
*   `rel`: Relationship type (e.g., `depends_on`, `supersedes`).
*   `valid_from` / `valid_to`: Bi-temporal logical timestamps (ISO8601).
*   `supersede`: Boolean flag. If true, automatically retracts existing active edges of the same type between the same nodes.

## 4. API Contract (Node-API FFI)

All mutations and queries are executed asynchronously via `tokio::task::spawn_blocking` to prevent blocking the V8 event loop.

### 4.1 Lifecycle
*   `open(opts: OpenOptions) -> Result<GenesisDatabase>`: Initializes the engine, acquires the OS-level file lock, loads the binary snapshot, and replays the JSONL WAL.
*   `compact() -> Result<()>`: Prunes retracted edges, writes a new temporary JSONL, atomically renames it to the main log path, and generates a new Bincode snapshot.

### 4.2 Mutations
*   `add_node(args: NodeInput) -> Result<NodeOutput>`: Inserts or updates a node and triggers an incremental impact refresh.
*   `add_edge(args: EdgeInput) -> Result<EdgeOutput>`: Validates Axiomatic Guards, inserts the edge, updates indices, handles supersession logic, and triggers impact refresh.
*   `retract_edge(id: String, at: Option<String>) -> Result<Option<EdgeOutput>>`: Marks an edge as logically deleted by setting `valid_to`.

### 4.3 Queries
*   `neighbors(seed: String, args: NeighborInput) -> Result<Vec<NeighborOutput>>`: Performs an N-hop BFS traversal. Supports filtering by direction (`in`, `out`, `both`), relationship types, and temporal state (`as_of`).
*   `cypher(query: String) -> Result<serde_json::Value>`: Executes a Cypher v0 query string.
*   `query(args: QueryInput) -> Result<Vec<EdgeOutput>>`: Direct edge filtering based on from/to/rel constraints.

## 5. Algorithmic Specifications

### 5.1 Incremental Impact Scoring
Instead of a full $O(N)$ recalculation on every mutation, the engine tracks the mutated node and performs a localized BFS to update only the downstream nodes affected by the change, reducing time complexity to $O(V_{affected} + E_{affected})$.

### 5.2 Axiomatic Governance Logic
Before an edge of type `supersedes` or `contradicts` is created, the engine evaluates the "Tier Score" of the source and target nodes based on their ID prefixes.
*   `MASTER--` / `FRAME--` = 1.0
*   `CONCEPT--` / `SPEC--` = 0.8
*   `FEAT--` / `ADR--` / `BLUEPRINT--` = 0.6
*   **Rule:** If `Tier(source) < Tier(target)`, the transaction is aborted with an Axiomatic Guard error.

## 6. Persistence & Safety Guarantee
*   **Crash Resilience:** The `compact()` function writes the new state to a `.tmp` file. Only after the file is fully flushed to disk is it renamed to overwrite the active log. The OS guarantees the atomicity of the `rename` operation, ensuring the database is never left in a corrupted state.
*   **Process Locking:** An OS-level lock file (`genesis-graph.lock`) containing the active PID prevents multiple host processes from concurrently mutating the same graph storage directory.
